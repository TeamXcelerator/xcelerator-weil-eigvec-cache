# xcelerator-weil-eigvec-cache

Consolidated, public cache of **CCM Weil-form eigenvectors** — the
smallest-eigenvalue eigenvector ξ of the Weil quadratic form — used
across Team Xcelerator's high-precision spectral-triple work.

> **Author:** Ronnie Andrews, Jr. (Team Xcelerator Inc.®)
> **License:** See `LICENSE`. Public for academic verification and
> reproduction; redistribution and commercial use restricted.

## What this is

The Weil eigenvector ξ is uniquely determined by three integers:

- `lambda_sq` (λ²) — the CCM prime-cutoff parameter
- `nmodes` (N) — the mode cutoff (vector length is `2N+1`)
- `prec` — the working precision, **in bits**

ξ is the smallest-eigenvalue eigenvector of the `(2N+1)×(2N+1)`
Weil-form matrix τ, found by inverse iteration and ℓ²-normalized so
`Σ ξ_j = √(ln λ²)`. It is **project-agnostic**: for a given
`(λ², N, prec)` it is the same vector no matter which experiment
computed it. That universality is why these caches can be consolidated
into a single shared repository.

This is a **distinct quantity** from the sibling caches:

- [`xcelerator-tau-cache`](https://github.com/TeamXcelerator/xcelerator-tau-cache)
  stores the τ-matrix (the operator); this repo stores its smallest
  eigenvector.
- The prolate eigenvalue cache stores a *different operator's*
  *eigenvalues*; this repo stores the *Weil operator's* *eigenvector*.

Each vector is generated by the Xcelerator Toolkit
(`xc-spectral::ccm::hp`) and stored as a deflated zip of the canonical
JSON form.

## Why it exists

ξ is expensive to produce: building τ (HP Gauss-Legendre integration +
prime-power sums at thousands of digits) followed by a dense LU
factorization and inverse iteration on a `(2N+1)×(2N+1)` matrix — the
LU is the dominant `O(N³)` cost, minutes to hours per config.
Recomputing ξ on every fresh cloud run wasted large amounts of compute.
This repo lets any run **fetch only the eigenvectors it needs** — by
deterministic URL, no clone required — and skip both the τ build and
the LU.

## Determinism

Caching ξ is only sound because the toolkit's eigenvector solve is
**bit-reproducible**: the inverse-iteration reductions
(`lu_solve`, the ℓ²-normalization, and the Rayleigh quotient) use a
fixed index-order fold rather than a runtime-ordered parallel
reduction, so the same `(λ², N, prec)` yields a byte-identical ξ on
every run and every machine. Files cached here are therefore safe to
deduplicate and share.

## Layout

Files are organized **precision-first, then by λ², then bucketed by
nmodes** in thousands — consistent with the sibling
[`xcelerator-gl-cache`](https://github.com/TeamXcelerator/xcelerator-gl-cache)
and [`xcelerator-tau-cache`](https://github.com/TeamXcelerator/xcelerator-tau-cache)
repos. No directory exceeds GitHub's web-UI listing limit and the path
is deterministically computable from `(λ², N, prec)`:

```
weil_eigvec_cache/
  prec{P}/
    lambda_sq{L}/
      nmodes{B}-{B+999}/
        weil_eigvec_lambda_sq{L}_nmodes{N}_prec{P}.json.zip
```

where:
- `P` = precision in bits (e.g. `3338` for HP-1000)
- `L` = λ² (the prime-cutoff parameter)
- `N` = mode count (vector length `2N+1`)
- `B` = `floor(N / 1000) * 1000` (the lower edge of N's thousand-bucket)

**Example:** the HP-1000 (`prec=3338`) Weil eigenvector for λ²=1000,
N=800 lives at

```
weil_eigvec_cache/prec3338/lambda_sq1000/nmodes0-999/weil_eigvec_lambda_sq1000_nmodes800_prec3338.json.zip
```

The **filename keeps the full `weil_eigvec_lambda_sq{L}_nmodes{N}_prec{P}`
key**, so each file is self-describing even if moved, and matches
exactly what the toolkit writes — no renaming needed on fetch.

### No byte-splitting

Unlike the τ-cache, ξ vectors are small (`2N+1` entries, ≲2 MB
compressed even at HP-1000/N=800), always well under GitHub's 100 MB
file limit. Every config is therefore a **single `.json.zip`** — there
is no `.partXX` tier, matching the GL-node cache.

### Path formula (for tooling)

```
weil_eigvec_cache/prec{P}/lambda_sq{L}/nmodes{floor(N/1000)*1000}-{floor(N/1000)*1000+999}/weil_eigvec_lambda_sq{L}_nmodes{N}_prec{P}.json.zip
```

## Fetching (no clone)

This repo is public, so any file is one raw URL, e.g.:

```bash
curl -sSL -O \
  https://raw.githubusercontent.com/TeamXcelerator/xcelerator-weil-eigvec-cache/main/weil_eigvec_cache/prec3338/lambda_sq1000/nmodes0-999/weil_eigvec_lambda_sq1000_nmodes800_prec3338.json.zip
```

The Xcelerator Toolkit does this automatically on a local cache miss
under `CacheMode::DynamicFetch` (the default): it probes the single zip
by deterministic URL and treats a 404 as a definitive miss (falling
through to a fresh compute), while retrying 429/5xx with backoff.

## File format

Each `.json.zip` is a deflate archive containing a single entry named
`weil_eigvec_lambda_sq{L}_nmodes{N}_prec{P}.json`: a JSON object
(`schema_version: 1`) with

- `lambda_squared`, `n_modes`, `precision_bits` — the cache key
- `weil_min_eigenvalue` — the smallest Weil eigenvalue ε_N, as a
  decimal string at the stated precision
- `xi` — a JSON array of `2N+1` decimal strings (the ℓ²-normalized
  eigenvector, `Σ ξ_j = √(ln λ²)`), at the stated precision

The Xcelerator Toolkit reads/writes this format directly and validates
it on load in two tiers:

1. **Structural** (always): vector length = `2N+1`, no NaN/Inf, and the
   `(λ², N, prec)` metadata match the requested key.
2. **Eigen-residual** (when the τ-matrix is in hand): the cached pair
   `(ξ, ε_N)` is verified against τ via `‖τξ − ε_N·ξ‖` below the
   working-precision floor — the strong integrity test that catches a
   structurally-valid-but-wrong vector. This is why the toolkit's cache
   check sits *after* the τ build.
